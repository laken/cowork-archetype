# CoWork + Claude in Chrome Reliability Briefing — 2026-03-09

## Reliability Signal

**Broadly reported, not isolated.** The CoWork + Chrome integration on macOS has multiple open GitHub issues spanning December 2025 through March 2026. The core connection problem — extension reports "not connected" despite correct setup — is the single most common complaint across [GitHub issues](https://github.com/anthropics/claude-code/issues?q=chrome+extension+connected), [X/Twitter threads](https://x.com/dguido/status/2015675676766916761), and the [Claude Help Center troubleshooting page](https://support.claude.com/en/articles/12902405-claude-in-chrome-troubleshooting). Community sentiment: frustrated but patient. Users on Max plans ($200/mo) are particularly vocal since browser automation is a headline feature.

No new Reddit threads or forum posts surfaced in the last 24-48 hours specifically about this issue, but the existing GitHub issues remain open and active.

## Failure Modes Observed

1. **"No Chrome extension connected" on startup** — Extension installed and enabled, native host running, socket exists, but MCP bridge never connects. Most common on macOS Sequoia + Apple Silicon. ([#29933](https://github.com/anthropics/claude-code/issues/29933), [#25151](https://github.com/anthropics/claude-code/issues/25151))

2. **Silent mid-session disconnects** — Extension drops connection during screenshot/scroll operations and enters unrecoverable exponential backoff (100+ reconnect attempts over ~2.5 hours). Requires full Chrome restart. ([#28013](https://github.com/anthropics/claude-code/issues/28013), [#26449](https://github.com/anthropics/claude-code/issues/26449))

3. **CoWork ↔ Claude Code conflict** — Both apps register native messaging hosts for the same extension ID. Chrome picks one; the other silently breaks. Everything *looks* correct but the socket format/location is wrong for the non-selected app. ([#29057](https://github.com/anthropics/claude-code/issues/29057), [#24539](https://github.com/anthropics/claude-code/issues/24539))

4. **MCP registry regression post-update** — CoWork's MCP client stopped connecting to bridge-type servers (mcp-registry, Claude in Chrome) after the ~Feb 13-14 auto-update to Desktop v1.1.3189. Red error banners at top of every session. ([#27492](https://github.com/anthropics/claude-code/issues/27492))

5. **Chrome Canary hijack** — When both Chrome and Chrome Canary are installed, browser tools connect to Canary even without the extension installed there. ([#31771](https://github.com/anthropics/claude-code/issues/31771))

6. **Auth/session mismatch** — MCP bridge rejects connections due to stale session tokens or cached identity state. Re-authenticating doesn't resolve. ([#24593](https://github.com/anthropics/claude-code/issues/24593))

## Updates Today

- **Claude in Chrome extension**: Recent updates include defaulting to Haiku 4.5 for speed, scheduled tasks, "follow a plan" workflow mode, and model selection (Haiku/Sonnet/Opus). A fix shipped for Chrome extension auto-detection getting stuck on "not installed" after running on a machine without local Chrome. No specific fix for the core connection reliability issues listed above. ([Release notes](https://support.claude.com/en/articles/12306336-claude-in-chrome-release-notes))

- **CoWork desktop app**: Windows support shipped with full feature parity. New Customize section groups skills, plugins, and connectors. Global and folder instructions added. No specific macOS reliability fixes mentioned in recent announcements. ([Releasebot](https://releasebot.io/updates/anthropic/claude))

- **Security note**: LayerX disclosed a CVSS 10.0 RCE vulnerability in Claude Desktop Extensions (DXT) — a single Google Calendar event can compromise a system. DXTs run without sandboxing with full host privileges, unlike Chrome extensions. ([LayerX disclosure](https://layerxsecurity.com/blog/claude-desktop-extensions-rce/))

## Trend

**First run — no prior baseline.** Based on issue timestamps, the core connection problems have been persistent since at least December 2025. The Feb 13-14 Desktop auto-update introduced a regression that remains unresolved. New features are shipping (scheduled tasks, model selection, Windows parity) but the fundamental connection reliability on macOS has not been publicly addressed with a fix. The Chrome Canary issue ([#31771](https://github.com/anthropics/claude-code/issues/31771)) is recent and suggests the connection routing logic remains fragile.

**Bottom line**: If you depend on CoWork + Chrome working reliably on macOS, expect intermittent failures. The workaround of restarting Chrome and CoWork in the correct order remains the primary mitigation.
