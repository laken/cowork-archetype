# CoWork + Claude in Chrome Reliability Briefing — 2026-03-10

## Reliability Signal

**Broadly reported, still unresolved.** The same core connection failures documented yesterday remain open with no targeted fix shipped. New issue [#32496](https://github.com/anthropics/claude-code/issues/32496) ("Chrome extension not working") filed as of this reporting period, indicating fresh reports continue to arrive. Dan Guido's X thread is now circulating as ["the internet's most authoritative troubleshooting thread"](https://x.com/dguido/status/2015675676766916761) — notable because it signals that community workarounds are becoming the de facto fix rather than official patches. A specific version-stamped report surfaced for v2.1.71 on macOS with a socket path issue at `$TMPDIR`, suggesting the problem is reproducible on current extension versions.

Community sentiment: **unchanged — frustrated, working around it.** No new forum threads or Reddit posts specific to the last 24 hours, but GitHub issue activity is continuous.

## Failure Modes Observed

Same set as yesterday, with one clarification:

1. **"No Chrome extension connected" on startup** — Persists on macOS Sequoia + Apple Silicon. ([#29933](https://github.com/anthropics/claude-code/issues/29933), [#25151](https://github.com/anthropics/claude-code/issues/25151))

2. **Mid-session disconnects after screenshot/scroll** — CDP session detaches, enters unrecoverable exponential backoff. Requires full Chrome restart. ([#28013](https://github.com/anthropics/claude-code/issues/28013), [#26449](https://github.com/anthropics/claude-code/issues/26449))

3. **CoWork ↔ Claude Code CLI conflict** — Competing native messaging hosts, incompatible socket formats. Only one can work at a time; the other silently fails. ([#29057](https://github.com/anthropics/claude-code/issues/29057)) — **Dan Guido confirmed on X this is the most common source of inexplicable failures.**

4. **MCP registry regression (post-Feb 13-14 auto-update to Desktop v1.1.3189)** — Red error banners on mcp-registry + Claude in Chrome at every session start. ([#27492](https://github.com/anthropics/claude-code/issues/27492)) — **Still open, no patch noted.**

5. **Chrome Canary hijack** — Browser tools route to Canary even without extension installed there. ([#31771](https://github.com/anthropics/claude-code/issues/31771))

6. **Auth/session mismatch** — Stale session tokens cause MCP bridge rejection. Re-auth doesn't fix. ([#24593](https://github.com/anthropics/claude-code/issues/24593))

7. *(New detail)* **Service worker goes idle during extended sessions** — Anthropic has acknowledged this: "the Chrome extension's service worker can go idle during extended sessions, which breaks the connection." Official mitigation: run `/chrome` and select "Reconnect extension."

## Trend

**Flat. No improvement vs. yesterday.**

The Feb 13-14 regression (MCP registry + Chrome bridge failure) is now ~25 days old with no public patch. New issues continue filing. The only movement is Anthropic adding the `/chrome reconnect` workaround to official docs — which confirms the service worker idle disconnect is real and recurring, but is a band-aid, not a fix.

The CoWork ↔ Claude Code conflict is increasingly well-documented publicly (Dan Guido thread going wide) but no architectural fix has shipped.

**Bottom line**: Connection reliability on macOS remains unreliable. Primary mitigations: (1) don't run CoWork and Claude Code CLI simultaneously, (2) remove Chrome Canary if installed, (3) use `/chrome → Reconnect extension` if tools stop responding mid-session, (4) full Chrome restart for unrecoverable backoff loops.
